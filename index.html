Salesforce Apex keywords

default,with sharing,with out sharing,inherited sharing Keywords

default

  • If the class is not declared as With Sharing or Without Sharing then the class is by default taken as Without Sharing.

    with sharing

  • If you declare a class as a With Sharing, sharing rules given to the current user will be taken into consideration and the user can access and perform the operations based on the permissions given to him on objects and fields. (field level security, sharing rules)

    with out sharing

  • If you declare a class as a Without Sharing then this apex class runs in system mode which means apex code has access to all the objects and fields irrespective of current users sharing rules, fields level security, object permissions.
  • Both inner class and outer classes can be declared as With Sharing.
  • If inner class is declared as With Sharing and top-level class is declared as without sharing, then the entire context will run in With Sharing context.
  • Outer class is declared as With Sharing and inner class is declared as Without Sharing then inner class runs in Without Sharing context only. (Inner classes don’t take the sharing properties from outer class).

    inherited sharing Keywords

  • Salesforce “Inherited Sharing” is a newly introduced modifier for class declaration. It’s surely better than Apex classes with no sharing declaration, as it mostly enforces WITH SHARING context, the official documentation screenshot describes Aura, VF, and Apex REST to auto enforce WITH SHARING context.
Context Inherited Sharing Implication
Aura WITH SHARING
Batch Apex WITH SHARING
Inbound Email Handler WITH SHARING
Scheduled Apex WITH SHARING
Trigger WITHOUT SHARING
Invocable Actions WITH SHARING
RESTful Apex WITH SHARING

  • 依赖

依赖(一条可能有方向的虚线)

依赖是两个事物间的语义关系，其中一个事物（独立事物）发生变化会影响另一个事物（依赖事物）的语义。一条可能有方向的虚线

关联

是一种结构关系，它描述了一组链，链是对象之间的连接。

  • 聚合
  • 组合

聚合(实线空心菱形，指向整体)

部分和整体的生命周期不一致，整体消失了，部分仍然存在，部分可以脱离整体存在

组合(实线实心菱形，指向整体)

部分和整体的生命周期一致，整体消失了，部分也消失，部分不可以脱离整体存在

关联多重度

  • 单项关联(一条有方向的实线)

泛化(实线空心箭头，指向父类)

泛化是一种特殊/一般关系，特殊元素(子元素)的对象可替代为一般元素(父元素)的对象。用这种方法，子元素共享父元素的结构和行为。

实现(虚线空心箭头，指向父类)

一种是在接口和实现类之间，另一种是在用例和实现它们的协作之间

防火墙技术

包过滤，应用代理网关和状态检测技术三个发展阶段

包过滤防火墙

  • 低水平控制 ：每个IP包的字段都被检查（源地址，目的地址，协议和端口）
  • 对用户完全透明，速度较快

应用代理网关防火墙

用代理网关防火墙彻底隔断内网与外网的直接通信，内网用户对外网的访问变成防火墙对外网的访问，然后再由防火墙转发给内网用户。

  • 优点是可以检查应用层，传输层和网络层的协议特征，对数据包的检测能力比较强
  • 缺点是难以配置，处理速度非常慢

关系模型中的基本术语

  • 关系：一个关系就是一张二维表，每个关系有一个关系名。
  • 元组：表中的一行即为一个元组，对应存储文件中的一个记录值。
  • 属性：表中的列称为属性。
  • 域：属性的取值范围。
  • 关系模式：对关系的描述称为关系模式，由关系名和其属性集合构成，关系模式的格式为：
  • 候选码（或候选键）：属性或属性组合，其值能够唯一地标识一个元组。
  • 主码（或主键）：在一个关系中可能有多个候选码，从中选择一个作为主码。
  • 主属性：包含在任何候选码中诸属性称为主属性。
  • 外码（或外键）：如果一个关系中的属性或属性组并非该关系的码，但它是另外一个关系的码，则称其为该关系的外码。
  • 全码：所有属性组是这个关系模式的候选码，称为全码。
  • 超码（超键）：一个包含码的属性集称为超码。

关系模型

关系模型由关系数据结构，关系操作集合和关系完整性约束三部分组成。

  1. 关系数据结构
    关系模型中数据的逻辑关系是一张二维表格。
  2. 关系操作集合（关系代数）
    关系模型给出了关系操作的能力。常用的关系操作有：选择，投影，连接，除，并，交，差等查询操作和增加，删除操作等。
  3. 关系完整性约束
    • 实体完整性：关系中主码的值不能为空或部分为空。
    • 参照完整性：外键必须对应上。
    • 用户定义的完整性：指用户对某一具体数据指定的约束条件进行检验。

范式

  • 第一范式(1NF) :不可分割的数据项，不能排除数据冗余和更新异常等问题，因为其中可能存在部分函数依赖。
  • 第二范式(2NF) :当切仅当R是1NF,且每个非主属性都完全函数依赖于候选码。可能存在传递函数依赖。
  • 第三范式(3NF) :当切仅当R是2NF,且每个非主属性都非传递函数依赖于候选码。可能存在主属性对码的部分依赖和传递依赖。
  • BC范式(BCNF) :不可分割的数据项，不能排除数据冗余和更新异常等问题，因为其中可能存在部分函数依赖。

1NF-消除非主属性对码的部分函数依赖->2NF-消除非主属性对码的传递函数依赖->3NF-消除主属性对码的部分和传递函数依赖->BCNF
